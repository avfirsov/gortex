# Temporal measurement harness — нужен ли вход-адаптер?

> Цель: на **реальном корпусе** получить числа, отвечающие на стратегический вопрос —
> достаточно ли «жадная эвристика (вход) + LLM-FP-фильтр (выход)», или реально нужен
> вход-адаптер (allow-list / нулевой discovery) сверх этого. Решение — по замеру, не по интуиции.

Принцип (см. обсуждение): **вход = recall** (сделать ребро видимым для движка), **выход = precision**
(убрать неверные). recall нельзя починить на выходе; precision дёшево не починить на входе. Поэтому
несущая пара — жадный вход + умный выход; allow-list/discovery в середине — оптимизация, и нужна
только если замер это показывает.

## Что меряем

Гоняем `gortex analyze` (daemonless, сборка из `feat/temporal-fork-all`) в трёх конфигурациях на
каждом репо:

| # | Конфигурация | Команда | Что даёт |
|---|---|---|---|
| 1 | **Baseline** — эвристика + built-in allow-list, без корп-списка | `temporal_orphans` (gate OFF) | broken_dispatch при «голой» жадной эвристике |
| 2 | **+ корп allow-list** | `temporal_orphans` (gate ON, есть `.gortex/temporal-allowlist.yaml`) | Δ broken_dispatch = recall, который дают корп-имена (non-env-named хелперы) |
| 3 | **+ выходной фильтр** | `temporal_verify` (нужен `llm.provider`) | confirmed / rejected / uncertain по каждому источнику |

## Как запустить

```bash
# из gortex (feat/temporal-fork-all): go build -o gortex ./cmd/gortex/  (нужен CGO)
# нужен jq в PATH; для шага 3 — настроенный llm.provider (ваш custom-provider)
export GORTEX_LLM_API_KEY=...                      # как в L1-прогоне
docs/temporal-compare/measure-temporal.sh /path/to/repo-a /path/to/repo-b ...
# результаты + сводка → ./temporal-measure/  (переопредели GORTEX_MEASURE_OUT)
```

Чтобы шаг 2 что-то показал — в каждом репо положи git-ignored
`.gortex/temporal-allowlist.yaml` с реально используемыми именами env-хелперов (см. `AGENTS.md`).
Без него шаг 2 = шаг 1 (Δ=0), и это само по себе сигнал.

## Как читать (правила решения)

### A. Нужен ли вход-адаптер (allow-list / zeroth)?

- **Δ broken_dispatch (шаг 1 → шаг 2) ≈ 0** → корп-имена ничего не добавляют к recall: **вход-адаптер
  НЕ нужен**, жадная эвристика покрывает. Нулевой discovery — тем более не нужен.
- **Δ broken_dispatch заметный** → есть non-env-named хелперы, которые эвристика не цепляет →
  вход-адаптер (allow-list, а дальше его автозаполнение нулевым discovery) **окупается**.

### B. Делает ли выходной фильтр реальную работу (precision)?

Смотри разбивку `temporal_verify` **по источнику** (`heuristic` / `convention` / `fuzzy` /
`env_default:*`):

- **rejected у `heuristic` высок** → жадная эвристика плодит FP → выходной фильтр **обязателен**
  (и это ок — он для этого). Если при этом confirmed тоже высок — recall ценный, фильтр окупается.
- **rejected у `heuristic` ≈ 0, confirmed высок** → эвристика почти всегда права → выходной фильтр
  **дёшев и почти не нужен**; allow-list лишь делает видимым то, что и так верно (маржинальная польза).
- **много `uncertain`** → модель/грунтинг слабоваты на вашем коде (тело хелпера вне репо) → стоит
  усилить грунтинг (call-site контекст) либо признать предел.

### C. Стоимость

`checked` — это размер выходной работы LLM на прогон. Если он велик и B показывает, что эвристика
в основном права, — вот тогда **нулевой discovery окупается**: запечь имена в конфиг один раз,
чтобы не гонять выходную LLM по ним каждый раз.

## Итоговая развилка

| Замер | Вывод |
|---|---|
| Δ≈0, reject(heuristic) низок | Хватает «жадная эвристика + дешёвый выход». Вход-адаптер и zeroth — **не строить.** |
| Δ заметный | non-env-named recall реален → **строить allow-list**, дальше zeroth для автозаполнения. |
| reject(heuristic) высок | Выходной фильтр несущий → **держать**; вход можно ужесточать по желанию. |
| checked велик + эвристика права | **zeroth discovery окупается** как cost-оптимизация. |

> Замечание: текущая эвристика «полу-жадная» (требует `env` в имени). Если хочется проверить
> гипотезу «максимально жадный вход + только выходной фильтр», нужен тумблер, снимающий это
> требование (эмитить speculative на любой `helper(x, "литерал") → dispatch`) — это маленькая
> доработка экстрактора, добавим, если замер по A окажется пограничным.
